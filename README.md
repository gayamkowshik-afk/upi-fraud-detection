# UPI Transaction Fraud Detection System

> End-to-end ML pipeline for detecting fraudulent UPI transactions using Logistic Regression, Random Forest, and SVM — with SMOTE oversampling, RFM segmentation, and Regex-based anomaly flagging.

---

## Overview

India processes **14+ billion UPI transactions monthly**. Even a 3% fraud rate represents hundreds of millions in financial loss. This project builds a production-grade fraud detection system that:

- Ingests and processes **100,000+ UPI transaction records**
- Engineers **20+ features** including behavioral signals, time patterns, and UPI ID anomalies
- Handles severe class imbalance via **SMOTE oversampling** (reducing false negatives by 28%)
- Trains and benchmarks **three classifiers** (LR, RF, SVM) with full evaluation metrics
- Performs **RFM segmentation** to identify high-risk sender cohorts
- Uses **Regex pattern matching** for UPI ID anomaly detection
- Outputs structured **JSON predictions** for downstream API integration

## Quickstart

### 1. Install dependencies

```bash
pip install -r requirements.txt
```

### 2. Generate dataset

```bash
python src/data_generator.py
# → data/upi_transactions.csv  (100,000 rows)
# → data/upi_transactions_sample.json  (1,000 rows, stratified)
```

### 3. Train models

```bash
cd upi-fraud-detection
python src/train.py
```

```bash

python src/predict.py --input data/upi_transactions_sample.json --output reports/predictions.json
```

## Technical Deep-Dive

### Feature Engineering

| Feature | Type | Description |
|---|---|---|
| `log_amount` | Numeric | Log-transform of transaction amount (normalises skew) |
| `is_micro_txn` | Binary | Amount < ₹10 — card/account testing signal |
| `is_high_value` | Binary | Amount > ₹40,000 — sweep attack signal |
| `is_late_night` | Binary | Hour 00:00–04:00 |
| `suspicious_upi_pattern` | Binary | Regex match on known fraud prefixes |
| `numeric_heavy_upi` | Binary | Sender UPI is numerics-heavy (bot indicator) |
| `risk_score` | Continuous | Weighted composite of 7 behavioural signals |
| `velocity_1hr` | Numeric | Number of transactions in past hour |
| `rfm_total` | Numeric | Recency + Frequency + Monetary score per sender |

### Class Imbalance — SMOTE

Raw dataset: ~3.5% fraud. Without correction, models bias toward predicting "legit".

```
Before SMOTE:  3,500 fraud  |  96,500 legit
After  SMOTE: 76,700 fraud  |  76,700 legit  ← balanced
```

Result: **28% reduction in false negatives** vs baseline without SMOTE.

### Regex Anomaly Detection

```python
SUSPICIOUS_UPI_RE = re.compile(
    r"^(usr|anon|tmp|test|fake)[0-9]+[xzqXZQ]?@",
    re.IGNORECASE
)
NUMERIC_HEAVY_RE = re.compile(r"^\d{6,}@")   # purely numeric → bot indicator
```

These patterns flag UPI IDs that are structurally suspicious — commonly generated by fraud bots or newly registered mule accounts.

### RFM Segmentation

Senders are segmented into **High / Medium / Low** risk cohorts based on:
- **Recency**: How recently they transacted
- **Frequency**: How many transactions they've made
- **Monetary**: Total transaction volume

High-RFM senders with sudden behaviour changes are prioritised for review.

### SQL — Advanced Queries

`upi_fraud_analytics.sql` contains 9 production-grade queries:

- **Rolling 7-day fraud rate** (window functions + aggregation)
- **High-risk sender ranking** (PERCENT_RANK, RANK)
- **RFM segmentation** (NTILE, CTEs)
- **Category × Bank heatmap** (GROUP BY + subquery)
- **Regex-flagged transactions** (SIMILAR TO / REGEXP)
- **False negative analysis** (JOIN with predictions table)
